# 🚀 Boostez vos performances avec l'IA — Formation Pratique 2026

> **Un parcours 100 % pratique pour transformer le cycle de vie logiciel (SDLC) avec l'Intelligence Artificielle Générative** — deux projets fil conducteur, une évaluation sécurité par scénarios réels et des démos interactives pour comprendre l'IA de l'intérieur.

## 📖 Le Pitch

Ce cursus intensif unifie les équipes techniques et métiers autour de **cas d'usage réalistes**. L'objectif n'est pas seulement de coder plus vite : c'est de transformer **chaque étape du cycle de vie logiciel** — de l'idéation au lancement — en exploitant l'IA comme un « pair » de très haut niveau.

La formation s'articule autour de **deux projets fil conducteur complémentaires** :

| | 🌱 **EcoTrack** | 🥐 **ZeroGaspillage** |
|---|---|---|
| **Produit fictif** | SaaS B2B de mesure de l'empreinte carbone IT (CSRD, Green IT) | Marketplace anti-gaspillage alimentaire (« Paniers Surprise » d'invendus) |
| **Écosystème** | Microsoft : C# 12, .NET 8, EF Core, SQL Server, SSRS/OLAP | Open-source : Python 3.12, FastAPI, PostgreSQL, React 18 + TypeScript |
| **Couverture** | Le cœur du développement : architecture → code → tests → CI/CD → BI | **Tout le SDLC** : idéation → marché → specs → build → lancement → V2 |
| **Volume** | 10 modules · 20 labs | 15 modules · 42 labs |

> 🧰 **Technologies et outils libres** : les stacks et outils cités partout dans ce support sont des **exemples populaires, jamais des exigences**. Chaque participant réalise les labs avec sa propre stack et l'outil IA de son choix (voir [la section Outils](#-outils--technologies--le-choix-vous-appartient)).

- **Usages avancés (5-8)** : Prompt Engineering (constructeur de prompt), Hallucinations & Grounding (détecteur de faits inventés), RAG (pipeline animé), Agents & Function Calling (simulateur de boucle agentique)

## 🎯 Philosophie & Acquis

1. **Context Grounding** — fournir le bon contexte (MCD, DDL, design system, lexique) est la clé d'une réponse IA pertinente.
2. **Orchestration d'Agents** — découper les problèmes complexes et les confier à des IA spécialisées (`.agent.md`).
3. **Infrastructure via « Skills »** — figer les conventions d'entreprise dans des fichiers de référence ingérés par l'IA (`.skill.md`, `copilot-instructions.md`).
4. **Collaboration transversale** — les livrables d'une équipe (Gherkin, OpenAPI) deviennent le contexte d'entrée de l'autre.
5. **Esprit critique** — vérifier, sourcer, tester : l'IA propose, l'humain décide (et reste responsable).

## 🔄 CI/CD

Le déploiement est automatisé via **GitHub Actions** (workflow [`.github/workflows/ci-cd.yml`](.github/workflows/ci-cd.yml)), déclenché à chaque `push` sur `main` (et exécuté aussi, sans déploiement, sur les Pull Requests) :

1. **`validate` (CI)** — installe Node.js et exécute [`.github/scripts/validate-site.js`](.github/scripts/validate-site.js), qui :
   - vérifie la **syntaxe** de tous les `<script>` inline de chaque page HTML de la racine (aucune page n'a de build, donc c'est la seule vérification statique possible) ;
   - vérifie que tous les **liens internes** (`href="*.html"` et redirections `location.href`) pointent vers un fichier existant du dépôt, pour détecter les liens brisés avant qu'un apprenant ne les rencontre.
2. **`deploy` (CD)** — ne se déclenche **que si `validate` réussit** et uniquement sur `main` (jamais sur une Pull Request) : empaquette le contenu du dépôt et le publie sur **GitHub Pages** via les actions officielles (`configure-pages`, `upload-pages-artifact`, `deploy-pages`).

Concrètement : une page cassée (script invalide ou lien mort) **bloque le déploiement** au lieu de partir en production silencieusement — avant ce workflow, GitHub Pages republiait automatiquement chaque push sans aucune vérification. Le site reste sans étape de build : le workflow ne fait que valider et publier tel quel.

Pour lancer la validation en local avant de pousser :
```bash
node .github/scripts/validate-site.js
```
